### 1. 原体験

**書くべきこと**：
- UDC2025 の経験を簡潔に（3段落以内）
- 以下の要素を必ず入れる：
  1. チームは「実行部隊」だった（テーマもデータも外側で決まっていた）
  2. グループLINEはあったが「作動していなかった」。情報共有の場の形骸化
  3. 負荷は「気がついた人」に集中し、しかし銀賞を獲った
  4. この構造は誰かの悪意ではなく、それぞれの立場の制約が絡み合った結果だった
  5. 外部権威（先生・コンテスト）への依存構造があった
- 「これが OPS-0 の出発点である」と明記

**トーン**：分析的に。特定個人の批判ではなく、構造の問題として書く。

### 2. 核心理念

**書くべきこと**：

**2.1 問題の定義**
- チームには「気がつく人」と「気がつかない人」がいる。注意力の向き方の違いであり、個人の優劣ではない
- 仕組みがないと、「気がついた人」が「やる人」になり、負荷が集中する
- 「気がつかない人」は何も引き受けなくても咎められず、無力化する
- 結果、最も注意深いメンバーから疲弊し、チーム全体の思考が止まる

**2.2 解決の方向**
- 「気がついたこと」をチームの共有財産に変換する仕組みを作る
- 標語「仕組みで人の善意に頼らない」
- これは「気がつく人」をなくすためではなく、その注意力を最も活かせる場所に集中できるようにする構造的防御

**2.3 核心の5カ条**
> 情報はチームの共有財産である。
> 権限はタスクとセットで移譲される。
> リーダーの主機能はエラー検知である。
> 理想形より及第点を定義する。
> チームは、自らの設計を自ら育てていく。

**トーン**：淡々と、しかし確信を持って。

### 3. 設計原則

**書くべきこと**：以下の5原則を、各1〜2段落で説明する。**実装例は書かない。指針のみ。**

| # | 原則 | 説明の要点 |
| :--- | :--- | :--- |
| 1 | **及第点の設計** | 理想ではなく「最低限どこまでやればOKか」を最初に決める。完璧主義による停滞と、過剰な献身による疲弊を防ぐ |
| 2 | **場の設計** | 単にツールを選ぶことではない。その場が何のためにあり、どう使われるべきかを共有し、「未整理のままでも発言していい」という及第点を最初に言葉にすること。これがなければ、どんなツールも沈黙する |
| 3 | **情報は共有財産** | 情報は忖度して加工する献上品ではない。未整理のままでも共有に値する。**原則として共有の場でやりとりする**。個人でのやりとりが必要な場合もあるが、その場合も「なぜ個人で話したか」と「結論」は共有財産としてチームに戻す |
| 4 | **権限はタスクと共に** | 権限のないタスクは責任のない作業。放任ではなく、ビジョンからの逸脱を検知する仕組みとセットで移譲する |
| 5 | **リーダーはエラー検知装置** | 日常の役割は「承認」ではない。情報の滞留・負荷の集中・ビジョンからの逸脱を検知し、問いかけによって介入する |

**トーン**：規範的に。ただし「〜すべき」ではなく「〜することがOPS-0の指針である」という書き方。

### 4. プロセス：PCAD の発展形

**書くべきこと**：
- WAIOS が提唱する PCAD（Purpose → Check → Action → Do）を継承・敬意することを明記
- OPS-0 の PCAD を各フェーズで説明

| フェーズ | 説明 |
| :--- | :--- |
| P（Purpose：目的） | 及第点を含む「何のための仕事か」をチームで定義する |
| C（Check：照合） | 現状と目的を照合し、ズレを可視化する。ふり返りとエラー検知を含む |
| A（Action：判断） | 可視化された情報に基づき、人が判断する。判断と責任は常に人にある |
| D（Do：実行） | 整った前提の上で、はじめて実行に移る |

- WAIOS が企業の業務プロセスとして実装したものを、OPS-0 はより小さなチームの、より日常的な会話の中で実践することを目指す

**トーン**：先行知見へのリスペクトと、独自発展の線引きを明確に。

**書くべきこと**：

- 各軸の定義

| 評価軸 | 問うこと |
| :--- | :--- |
| 健全性 | プロセスは構造として機能していたか。特定の誰かに負荷が集中していなかったか |

| 学び | 個人・チームとして何を得たか。次のプロジェクトに継承できるか |

- これらの軸は「評価」のためではなく、次のプロジェクトをより良くするための「材料」である

**トーン**：道具的に。これは善悪を裁く物差しではない。

### 7. 限界と戒め

**書くべきこと**：

**7.1 未検証であること**
- OPS-0 は個人の経験と内省から導かれた仮説であり、多様な環境での検証を経ていない

**7.2 適用範囲の限界**
- 現状は小規模チーム（3〜5人程度）を想定。大規模プロジェクトへの適用は未検証

**7.3 触媒は人ではなく、行動にある**
- OPS-0 は、「場を提案する」「最初に共有する」「及第点を言葉にする」「問いかける」という行動を、チームの誰かが最初に取ることを必要とする
- これは特別な才能ではなく、意図と少しの勇気の問題である
- OPS-0 は、この行動を「触媒」と呼ぶ。しかしそれは特定の人材を選び出すためではなく、やがて誰がやったかも意識されなくなることを目指すためである

**7.4 仕組み化のリスク**
- 整えすぎが即興性・偶発性・遊びを失わせる
- OPS-0 は、整えすぎないための自制もまた、設計の一部とする

**7.5 OPS-0 は権威ではない**
- OPS-0 のどの指針よりも、チーム自身の納得感が上位にある
- チームは OPS-0 の指針を、納得できないなら修正し、捨て、独自のものに置き換える権利を持つ
- OPS-0 の作成者もまた、外部権威に従わざるを得なかった構造の中にいた。だからこそ、OPS-0 は自らが新たな「先生」になることを拒否する
- チームが OPS-0 を修正し、破棄し、独自のものに置き換えること——それは OPS-0 の失敗ではなく、OPS-0 の成功である

**7.6 AI と人間の境界**
- AI は思考を補助する道具であり、判断と責任は常に人が負う
- AI が「考えさせる」ツールになることと、人間が「考えなくなる」ことの境界を常に意識する

**7.7 使われ方への戒め**
- 及第点は「導入しないこと」も含む
- 「未整理の共有」は共有である
- 最初は一つだけでよい
- モデルは叩き台であり、権威ではない（7.5 を再掲してもよい）

**トーン**：誠実に。正直に。モデルを守るためではなく、使う人を守るために書く。

### 8. 結び

**書くべきこと**：
- OPS-0 は完成していない。そして完成を目指すものでもない
- これはチームが自分たちの構造を自分たちで見つめ、直し、育てていくための道具箱であり、共通言語であり、戒めである
- 最終目的は、チームが OPS-0 を不要にすること。チームが自らの言葉で課題を語り、自らの設計を育て始めた時、OPS-0 はその役割を終える
- 「整えすぎる前に、動き出せ」
- このマニフェスト自体も、書き換えられることを前提としている

**トーン**：開かれた感じで。閉じな
